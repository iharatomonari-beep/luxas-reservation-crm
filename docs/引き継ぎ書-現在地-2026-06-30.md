## 3. 絶対に守る不変条件・注意点

1. **★店舗解決の橋**：アプリ `Store.id`（"store-shibuya" 等）＝ DB `stores.code`。**uuid化禁止**（全テーブルの店舗解決が一斉に壊れる）。新店舗をDBに足すときも code にアプリの Store.id を入れる。
2. **組織は B-min**：tenants/areas/stores の読みは localStorage 据え置き（org mapper 不使用）。完全DB化(B-full)は将来必要なら別スライス。
3. **顧客＝個人情報**：実データは §5 #6/#8 後。今は架空のみ。
4. **メニューは店舗別**（profile.storeIds）。1店舗に全377は出ない（渋谷は84が正しい）。
5. **★空になり得る列はDBで必ず NULL 許容**（room_id事件の教訓）。
6. **実データはコード管理対象外**：カタログ・シフト・7店舗・RPC・列追加は **Supabase に適用済み＝gitには入らない**。コード（mapper/SQL雛形/docs）と実DB状態は別管理。

## 6. 進め方のルール（厳守）

- 各機能とも **設計提案 → ユーザー承認 → 実装**。1機能ずつ。
- **SQL・DB・破壊的変更・外部依存追加・実顧客データ投入はオーナー承認後**。SQLは1つずつ pbcopy でオーナーが Supabase SQL Editor で実行。
- 実装時は **lint/build を必ず通す**。新テーブルは DDL+RLS+seed を専用SQLファイルに。参照ありは legacy_id→uuid 解決＋round-trip 列。
- 完了ごとに `docs/sessions/` に追記し、本書・master-plan を更新。
