## 追記: 「会計→リロードで予約が消える」の真因（第2弾・自己再現で特定）
- 初回修正（localStorage再ハイドレートのlocal限定＝fix#1）後も再現。Claude自身が**本番/認証セッションのページ内JS＋コンソール直読み**で実フローを再現しトレース。
- **真の主因は DB削除ではなく「描画クラッシュ」**: `normalizeReservationForDisplay` が `normalizeText(reservation.serviceMenuId/staffId/roomId)` を呼び、`normalizeText` が `value.trim()`。予約に参照IDが欠けると `undefined.trim()` → `normalizedReservations` useMemo が例外 → **台帳全体が描画失敗＝予約が消えたように見える**。ネットワークでDELETEが出なかったのはこのため（削除ではなく描画失敗）。
- 欠損が起きる経路: オンライン予約RPC(`create_online_booking`)は profile/legacy_id 無しで insert するため、fromRow 復元時に serviceMenuId 等が undefined。
- 修正（コードのみ・型/RLS/DDL変更なし）:
  - `utils.ts` `normalizeText(value?: string|null)` を null安全化（`(value??"").trim()`）。
  - `reservation-mapper.ts` `fromRow` で `serviceMenuId/staffId/roomId` を必ず文字列(`?? ""`)に。
  - fix#1（localStorage再ハイドレートのlocal限定）は維持。
- **Claude自身が実ブラウザの台帳で検証**: 正常予約→リロードで `persist prev=1→1`（保持・削除なし）。欠損予約も含め2件→リロードで `persist prev=2→2`（**クラッシュ無し・両方保持**）。
- 診断ログ（`[diag] boot/resync/persist/DELETE`）は再確認用に**一時的に残置**。確認後に撤去予定。
- 注: ユーザーの再現が直らなかった一因は**ブラウザの旧コードキャッシュ**の可能性大（boot ログで判別可・ハードリロード推奨）。

## 追記: 「予約が消える」**真の根本原因＝確定・解決**（6/28）
- 長時間の切り分けの末、実DB再現で確定。`[supabase] sync reservations failed {}` の正体は2段階で誤誘導されたが、最終真因は:
  - **通常の施術ブース予約は `roomId` が空**（T011でブースは容量ベース＝個室以外は部屋を固定しない。`reservation-ledger.tsx`「通常ブースは空」）。

  - → だが **`reservations.room_id` が NOT NULL** だったため **code 23502（null violates not-null）でINSERT拒否** → DBに入らず消える。
  - 私のREST検証は実在 room_id を入れていたため成功＝この差分が核心。EXCLUDE(23P01)は別件の確認。
- **修正（DDL 1行・ユーザー承認のうえ適用）**: `alter table public.reservations alter column room_id drop not null;`。アプリの容量ベース・ブース設計に合致。room_id=null 行は room重複EXCLUDE対象外（null distinct）＝スタッフ重複のみ禁止で正しい。`supabase/schema.sql` も room_id を NULL許容に更新。
- 実DBで `room_id=null` INSERT が **201成功**を確認。ユーザーがUIで作成→リロードで**残存を確認（解決）**。
