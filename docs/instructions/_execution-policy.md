# Claude Code 実行ポリシー（常設ルール）

このファイルは、**Claude Code がこのプロジェクトで作業するとき、毎回いちばん最初に読む常設ルール**です。今回限りではなく、今後このプロジェクト全体で継続的に使います。

---

## 1. 最優先ルール

- Claude Code は、作業開始時に**必ず `docs/instructions/_execution-policy.md`（このファイル）を最初に読む**。
- **`_execution-policy.md` を読まずに `_index.md` や個別タスクを実行してはいけない。**
- 次に `docs/instructions/_index.md`（実行キュー）を読む。
- `_index.md` に記載された**未実行タスクを上から順（最小番号から）に処理する**。
- ただし、**すべてのタスクでユーザー確認を求めてはいけない**。
- 各タスクに付与された**安全レベル（SAFE / CAUTION / STOP）に応じて、自動実行・確認停止・実行停止を判断する**。

---

## 2. 基本方針

- SAFE タスクは確認なしで自動実行する。
- CAUTION タスクは実行前に停止して確認を求める。
- STOP タスクは実行せずに停止する。
- 各タスクは1つずつ処理する（複数をまとめて実装しない）。
- 各タスク完了後に `_index.md` の状態を更新する。
- エラーが出たら次に進まない。
- 仕様判断が必要な場合は停止する。
- 実機確認が必要な場合は停止する。
- 指示にないファイルは変更しない。
- 大規模リファクタリングはしない。
- 失敗を自己判断で大きく修正しない。
- 変更内容と検証結果を必ず報告する。

---

## 3. 安全レベル

各タスクは SAFE / CAUTION / STOP のいずれかを持つ。タスクファイルの記載を優先し、記載が無い・内容と矛盾する場合は**より安全側（SAFE→CAUTION→STOP の順に厳しい）に倒す**。

### SAFE：確認なしで自動実行してよい

対象:
- docs 配下の追記・修正
- `docs/instructions/_index.md` のステータス更新
- `docs/instructions/tasks/` 配下のタスク整理
- 調査結果の整理
- ファイル一覧確認
- grep / search によるコード確認
- 型や定数の読み取り
- 既存コードを変更しない調査
- lint / build / typecheck の実行
- 実行ログや検証結果の記録
- 明確に指定された1ファイルだけの軽微な docs 修正

SAFE のルール:
- 実行前確認は不要。
- 完了後に、**変更ファイル・検証結果・次のタスク**を簡潔に報告する。
- エラーがなければ次の未実行タスクへ進む。

### CAUTION：実行前に停止して確認する

対象:
- TypeScript / React / Next.js のコード変更
- 予約台帳、予約作成、顧客管理など主要画面のロジック変更
- localStorage の保存形式変更
- 既存の型定義変更
- 新しい設定ファイル追加
- 新しいデータ構造追加
- 複数ファイルにまたがる変更
- UI の見た目が変わる変更
- 動作仕様が変わる変更
- lint / build / typecheck の失敗を直すためのコード修正
- タスク指示に曖昧さがある場合
- 既存仕様書と実装が食い違っており判断が必要な場合

CAUTION のルール:
- 実行前に停止する。
- 以下の形式で確認を求める。
  - タスクID
  - 目的
  - 変更予定ファイル
  - 変更内容
  - 想定リスク
  - 検証方法
  - ユーザーに確認したい判断
- **ユーザーが許可するまで実行しない。**
- 許可が出た場合のみ、そのタスクだけ実行する。
- **判断を仰ぐときは必ず「選択肢」の形で提示する**（後述「判断を仰ぐときのルール」を厳守）。

### 判断を仰ぐときのルール（常設）

ユーザーに判断・選択を求めるときは、**必ず選択肢（A / B / C …）の形で提示する**。自由記述だけで漠然と尋ねない。

- 各選択肢に「内容・メリット/デメリット・リスク」を簡潔に添える。
- **推奨案を1つ明示**し、理由を1行で書く（無指定時はその推奨案で進めてよい）。
- 「どうしますか？」のような開放質問だけで終わらせない。必ず具体的な選択肢を並べる。
- 選択肢が2〜4個に収まらない複雑な判断は、論点ごとに分けてそれぞれ選択肢化する。
- 可能なら選択用UI（AskUserQuestion）で提示する。文章で出す場合も A/B/C の箇条書きにする。

### STOP：実行せず必ず停止する

対象:
- ファイル削除
- 既存データ削除
- DB マイグレーション実行
- Supabase 接続設定の変更
- `.env` / `.env.local` の中身閲覧または変更
- 認証・権限・RLS の変更
- `package.json` の依存関係追加・削除
- 大規模リファクタリング
- 仕様未確定のまま DB 構造を変える作業
- 実機確認が必要な作業
- 予約データ・顧客データの破壊可能性がある作業
- 指示にないファイルを変更する必要が出た作業
- 本番データ・個人情報・認証情報に触れる作業

STOP のルール:
- 実行しない。
- 以下を報告して停止する。
  - タスクID
  - 停止理由
  - 何が危険か
  - ユーザー確認が必要な判断
  - 次に選べる選択肢

---

## 4. 連続実行ルール

- SAFE タスクは連続して自動実行してよい。
- CAUTION タスクに到達したら、その時点で停止して確認を求める。
- STOP タスクに到達したら、実行せず停止する。
- エラーが出たら停止する。
- `build` / `lint` / `typecheck` が失敗したら停止する。
- 失敗を自己判断で大きく修正しない。
- 失敗原因を報告し、次の指示を待つ。
- 指示にないファイルは変更しない。
- 大規模リファクタリングはしない。
- 各タスク完了後、`_index.md` の状態を更新する。
- 完了したタスクは「完了」、停止したタスクは「停止」、判断待ちは「保留」とする。

---

## 4.5 自己解決・継続のルール（毎回の確認を減らす）

毎回ユーザーに確認を求めて止めると負担が大きい。次のルールで**できる限り自分で解決して進める**。

### 包括承認（セッション内の標準）
- ユーザーが「すべて実行」「未実行を全部」「進めて」等と言ったら、それは**そのセッション内の包括承認**とみなす。以後、同種の CAUTION タスクは**1件ごとの実行前確認を省略**して実装まで進めてよい（実装→lint/build→状態更新→次へ）。
- 包括承認中でも、**STOP 対象（削除・DB・env・認証・依存変更・指示外ファイル等）だけは必ず止める**。

### 判断は止めずに「推奨で進めて報告」
- 判断が必要な点は、**推奨案を1つ選んで実装し、選んだ理由を報告**する（止めない）。選択肢の提示が要るのは、推奨が決められない・後戻りが高コストな場合のみ。
- 仕様に既定値が書いてある判断（例: 既定担当＝その日先頭の出勤スタッフ）は、**そのまま採用**して進める。

### 実機確認は「ためずにバッチ」
- 実機確認が必要なタスクは、**実装＋lint/build まで進め「実行中（実機確認待ち）」**にして次へ進む。1件ごとに止めない。
- 区切りのよいところで**まとめて確認を1回依頼**する（「ここまでの N 件を確認してください」）。

### エラー・失敗は自己解決を試みる
- lint / build / 型エラーは、**指示の範囲内なら自分で原因を直して再検証**する（未使用import削除・型不一致の修正など軽微なものは確認不要）。直せた場合は確認を求めない。
- 範囲内で直せない・原因が仕様判断に及ぶ場合のみ停止して報告。
- ツール呼び出しの形式ミス等の自分起因の失敗は、**黙って正しい形式で再実行**する（ユーザーに確認しない）。
  - 既知の不具合: ツール呼び出しを誤った書式で出すと**何も実行されず「止まったように見える」**。必ず正規のツール呼び出し書式を使い、反応が無い＝自分の書式ミスを疑って即座に正しい書式で出し直す。

### 報告は簡潔に
- 各タスク完了後は結果のみ簡潔に。確認を求めるのは包括承認の範囲外・STOP・まとめ実機確認のときだけ。

---

## 5. タスクファイルの必須項目

各タスクファイル（`docs/instructions/tasks/` 配下）には必ず以下を書く。

- タスクID
- タスク名
- 安全レベル：SAFE / CAUTION / STOP
- 目的
- 背景
- 対象ファイル
- 変更してよいファイル
- 変更してはいけないファイル
- 実行内容
- 検証方法
- 完了条件
- 停止条件
- 完了後の報告内容

---

## 6. 完了報告形式

各タスク完了後は以下を報告する。

- 完了したタスクID
- 実行した内容
- 変更したファイル
- 実行した検証
- 検証結果
- `_index.md` の更新内容
- 次に進んだタスクID、または停止理由

---

## 7. 停止報告形式

停止した場合は以下を報告する。

- 停止したタスクID
- 安全レベル
- 停止理由
- 危険点
- ユーザー確認が必要な判断
- 次に選べる選択肢
- 残っている未実行タスク
