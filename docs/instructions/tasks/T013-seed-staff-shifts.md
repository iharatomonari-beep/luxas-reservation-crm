# タスクID：T013
# タスク名：仮スタッフ8人＋2ヶ月分のシフト（週5・9時間・土日厚め）を生成

## 安全レベル
CAUTION（モックデータ・初期データ生成。実行前にユーザー確認）

## 目的
動作確認用に、仮スタッフ8人と、各自週5日・1日9時間のシフトを 2026-06-13 から2ヶ月分用意する。曜日は固定せず、毎日まんべんなくスタッフがいて、かつ土日を厚めにする配分にする。

## 背景
- シフトは per-date（`StaffShift.workDate`）。週5を2ヶ月分＝1人あたり多数の日付エントリになるため、**ベタ書きせず生成ロジック（関数）で `initialShifts` を作る**こと。
- 休憩は空（T006方針）。`breakStart`/`breakEnd` は `""`。
- 既存の `initialStaff` を活かしつつ、合計8人（`isActive: true`）にする。既存予約が参照する staffId は壊さない（迷えば停止）。
- 古い localStorage が新データを上書きする問題があるため、staff/shifts も自動再シード対象にする（下記）。

## スタッフ8人の設計（時間帯と休み曜日）

| 通し | 勤務時間(9h) | 休み曜日 | 出勤曜日 |
|---|---|---|---|
| 1 | 10:00–19:00 | 土・日 | 月火水木金 |
| 2 | 10:00–19:00 | 月・火 | 水木金土日 |
| 3 | 10:00–19:00 | 火・水 | 月木金土日 |
| 4 | 12:00–21:00 | 水・木 | 月火金土日 |
| 5 | 12:00–21:00 | 木・金 | 月火水土日 |
| 6 | 14:00–23:00 | 月・金 | 火水木土日 |
| 7 | 14:00–23:00 | 月・木 | 火水金土日 |
| 8 | 14:00–23:00 | 火・金 | 月水木土日 |

- 結果の1日あたり出勤人数: 月5・火5・水6・木5・金5・**土7・日7**（土日厚め、平日も5〜6人）。各自ちょうど週5日。
- **定休日（週2日）は各スタッフ固定**。上表の「休み曜日」は毎週同じ曜日で固定し、週ごとに変えない。

## 実行内容
1. `initialStaff` を合計8人（`isActive: true`、`sortOrder` 1..8）にする。既存スタッフがいれば id/名前を維持して上の表の役割に割り当て、足りない分は仮名（例「スタッフ6」）で追加。
2. `initialShifts` を**生成関数**で作る:
   - 対象期間: `2026-06-13` 〜 その2ヶ月後（`2026-08-13`）までの各日。
   - 各スタッフについて、その日の曜日が「出勤曜日」に含まれる場合のみシフトを1件作る。
   - `startTime`/`endTime` は上表の9時間枠。`breakStart=""`/`breakEnd=""`。`isActive: true`。`id` は一意（例 `shift-<staffId>-<YYYYMMDD>`）。
3. `local-storage.ts` の `SEED_RESET_TOKENS` に次を追加し、古い localStorage を初期データへ強制再シードする:
   - `"luxas-master-staff": "2026-06-13-staff-8"`
   - `"luxas-master-shifts-v2": "2026-06-13-shifts-2mo"`
4. 生成結果が過大にならないか確認（8人×約43日＝数百件程度は許容）。
