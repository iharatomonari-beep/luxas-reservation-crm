# 機能オン/オフ層（外販カスタマイズ基盤）設計案

作成: 2026-06-30 / ステータス: 提案（未実装）。実装着手はオーナー承認後。

## 0. 目的

自社利用で固めたあと、他社・他店舗へ外販する。その際に**店舗（会社）ごとのニーズに合わせて機能の要不要をチェックボックスで選び、コードを分岐（フォーク）せずに納品**できるようにする。理想は「全機能にチェックボックスが付き、店舗単位でオン/オフ」。

## 1. 現状の結論（なぜ層を足すか）

- いまは**固定ベース寄り**。上部メニューは `top-menu.tsx` にべた書き、各ページはファイル単位ルートで、機能フラグの共通フレームワークは無い（オン/オフは `onlineReservationEnabled` の1つだけ）。
- ただし可変化の**土台は揃っている**：店舗ごとの `StoreSettings`（Supabase保存）、マルチテナント（テナント→エリア→店舗）とデータの店舗スコープ、レジストリ方式の前例（`migration-config`）、オン/オフ実装の実績（オンライン予約）。
- よって**作り直しではなく、既存の上に薄い制御層を1つ足す**方針が最適。既存機能のコードには基本手を入れない。

## 2. 設計（6コンポーネント）

### (1) 機能カタログ `features/platform/feature-catalog.ts`（新規・単一の真実）
切替可能な機能を一覧定義する。1エントリ＝1機能。
```
type FeatureDef = {
  key: string;            // "retail", "checkout", "mail-edm" ...
  label: string;          // 表示名（管理UI・メニュー）
  navGroup: string;       // 所属メニューグループ（reservations/daily/...）
  routes: string[];       // このキーが支配するルート（ガード対象）
  dependsOn?: string[];   // 依存する他機能キー（会計→メニュー 等）
  core?: boolean;         // true=常時ON・切替不可（固定コア）
  defaultOn?: boolean;    // 既定のオン/オフ（未指定はtrue）
};
```
- メニューやページを増やしたら、ここに1行足すだけで「切替対象」に載る。

### (2) 店舗ごとの有効機能設定（保存先は既存の器を流用）
`StoreSettings` に `enabledFeatures?: Record<string, boolean>` を追加。
- **保存先はすでに店舗単位でSupabaseにある**（新テーブル不要）。
- 白ラベル向けに、テナント既定（テナント単位のプリセット）→ 店舗で上書き、という2段にできる（P3）。

### (3) リゾルバ `isFeatureEnabled(key, ctx)`（新規・純関数）
実効オン/オフを解決する。
- `core` は常時ON。
- それ以外＝ 店舗設定 ?? テナント既定 ?? カタログ `defaultOn` ?? true。
- **依存の伝播**：`dependsOn` のどれかがOFFなら、その機能も実効OFF（グレーアウト理由を返す）。

### (4) メニュー絞り込み（`top-menu.tsx` 改修）
`NAV_GROUPS` をカタログ＋リゾルバで**フィルタ**して描画。無効な項目は非表示、全項目が無効なグループはグループごと非表示。既定（全ON）なら**今と同じ表示**＝非破壊。

### (5) ルートガード（`app/dashboard` レイアウト or 共通ラッパ）
URL直打ちで無効機能に入れないよう、現在ルートに対応する機能が無効なら `/dashboard`（または「この機能は無効です」画面）へリダイレクト。1枚の薄いガードで全ページをカバー。

### (6) 管理チェックボックスUI `店舗情報 > 機能設定`（新規画面）
グループ別にチェックボックスを並べ、店舗ごとにオン/オフを保存。依存関係を表示（例：会計をOFFにすると 支払・レジ/経営指標 も使えなくなる旨）。owner/管理者のみ。納品時はここで店舗に合わせて設定。

## 3. 固定コア vs 可変モジュールの線引き（初期案）

### 固定コア（`core:true`・常時ON・チェック不可）
- 認証／ログイン
- 組織（テナント／エリア／店舗）・店舗設定
- 予約台帳のコア（予約作成・予約一覧）
- 顧客管理のコア（顧客登録・検索）

### 可変モジュール（チェックボックスで切替）
- 物販（物販商品・物販販売・物販カテゴリ）
- 会計／支払・レジ
- 日次管理（打刻・レジ金・開閉店・売上日報）
- メール管理（定型文・eDM・シンプルeDM）
- 経営指標（各帳票・日次集計・スタッフ成績）
- オンライン予約（公開ページ・スタッフ別ロック）
- EPARK掲載設定 ／ セット商品 ／ 会計アイテム
- タグ類（顧客／予約ルート／施術カルテ）
- 顧客マージ ／ 顧客フル検索
- シフトの高度機能（ひな型・パターン・月間）

## 4. 依存関係（例・カタログの `dependsOn` に落とす）
- 会計 → メニュー（商品）
- 支払・レジ／経営指標／日次集計 → 会計
- 物販販売 → 物販商品
- オンライン予約公開 → メニュー＋スタッフ＋シフト
- eDM → 顧客＋メール定型文

## 5. 白ラベル（外販）で併せて効くもの
- ブランディング：`StoreSettings` に既にある storeName／storeShortName／背景色／HP画像等を活用（ロゴ差し替え等はP3で拡張）。
- データ分離：テナント／店舗スコープ＋RLSで実装済み。
- 納品物＝「機能チェック設定（店舗別）＋ブランド設定」。コードは共通のまま。

## 6. 実装フェーズ（非破壊・段階的）
- **P1**：カタログ＋リゾルバ＋`StoreSettings.enabledFeatures`＋メニュー絞り込み。既定は**全ON＝現状と同じ挙動**（回帰ゼロ）。
- **P2**：管理チェックボックスUI＋ルートガード。
- **P3**：テナント既定（白ラベルのプリセット）＋ブランディング拡張。
- **P4**：依存バリデーション＋無効時のUX（グレーアウト・理由表示）。

## 7. リスク・注意
- 依存関係を無視してOFFにすると壊れて見える → リゾルバで依存伝播＆UIで明示。
- 固定コアを誤ってOFFにできないよう `core` はUIから外す。
- 既定を全ONにして段階導入すれば、既存の自社利用に影響を出さずに入れられる。

## 8. 規模感
既存機能の改修はほぼ不要（上に載せる制御層）。工数の中心はカタログ定義とメニュー/ガード/管理UIの配線。P1だけでも「メニューを店舗設定で出し分け」が実現し、外販の骨格になる。

## 9. 設計レビュー反映（外販品質の追補）

外部レビュー（ChatGPT）で妥当な指摘があり反映する。要旨：メニュー/ルートは「見た目の制御」で、外販・マルチテナントSaaSの本丸は**サーバー側（RPC/RLS）の強制**。以下を設計の必須要素として追加する。

### 9.1 強制は4層（どこが見た目・どこがセキュリティか）
| 層 | 役割 | 性質 |
|---|---|---|
| メニュー表示 | 使わない機能を出さない | 見た目（利便性） |
| ルートガード | URL直打ちを弾く | 見た目寄り（回避可能性あり） |
| **API/RPC 実行拒否** | 無効機能のRPC/APIを実行させない | **セキュリティ本丸** |
| **RLS（行レベル）** | データそのものの越境防止 | 既存＝維持・強化 |

→ 表示・ルートだけだと「画面を隠したがRPCは動く」状態になり得る。**外販前に API/RPC ガードまで入れて初めて外販品質**。自社利用（単一テナント）の段階では表示＋ルートで十分。

### 9.2 機能フラグ と 権限ロール は別軸（混ぜない）
- **機能フラグ**＝「この店舗/会社でこの機能を使うか」（契約・ニーズ）。
- **権限ロール**＝「この人（owner/admin/staff）がこの操作をしてよいか」。
- 実効可否 ＝ `機能がON` **かつ** `ロールが許可`。カタログ（機能）とロール表（権限）は別テーブル/別定義にし、リゾルバで両方を掛け合わせる。既存の owner限定RPC（顧客マージ等）は「ロール軸」で、機能フラグとは独立。

### 9.3 フラグの粒度＝テナント（プラン）＋店舗（個別）
- **テナント（会社）＝エンタイトルメント/プラン**：外販先が「買った機能セット」。ここが上限。
- **店舗＝プラン内の個別オン/オフ**：同一会社でも店舗ごとに使う/使わないを調整。ただし**プランに無い機能は店舗でONにできない**（上限を超えない）。
- area は原則フラグ単位にしない（必要になったら店舗のグルーピングとして後付け）。
- 保存：テナントのプランは tenant 単位（tenant settings / 専用列）に置き、**RPC/RLSから参照できる場所**に持つ（StoreSettings の jsonb だけに閉じない）。店舗の個別オン/オフは StoreSettings で可。→ サーバー側で `feature_enabled(store, key)` を判定できる形にする。

### 9.4 リスク別に段階適用（過剰実装を避ける）
すべてを一律にRPCガードするのではなく、リスクで優先度をつける：
- **必須（先にやる）**：書き込み/副作用/越境の恐れがある機能（会計・マージ・メール送信・CSV入出力・オンライン公開）。
- **後回し可**：自店データの読み取りのみ（経営指標の閲覧等）。画面を隠せば実害は「自店の自データを見る」程度。

### 9.5 フェーズ改訂（サーバー強制を明示）
- P1：カタログ＋リゾルバ＋店舗設定＋メニュー絞り込み（既定=全ON・非破壊）。※自社利用はここまでで足りる。
- P2：管理チェックボックスUI＋ルートガード。
- P3：テナント＝プラン（エンタイトルメント）＋店舗＝個別上書き。ロール軸との掛け合わせ。
- **P4（外販前の必須ゲート）：API/RPC 実行ガード＋RLSとの整合＋監査ログに「機能利用」を記録**。ここを越えて初めて外販可。
- P5：白ラベル（ブランド）拡張・納品パッケージ化。

### 9.6 結論（レビューの受け止め）
- 方向性・非破壊・段階導入は妥当（P1として合格）。
- ただし現状の記述は「表示＋ルート」中心で、**サーバー強制・フラグとロールの分離・粒度（テナント/店舗）**が曖昧だった＝本追補で補正。
- 実装着手の前に、本追補を含めてもう一度**設計レビュー**を挟む（特に §9.1〜9.3）。いきなり実装しない。

### 9.7 RLS/RPC 実効監査（外販前の必須ゲート・断定しない）
「RLSは設定済み」と「正しく効いている」は別。外販前に実効監査を行う（§5 #8 とセットで）。現時点でコードから確認できた事実と、要監査項目を分ける。
- 確認できた事実（良好な兆候）：フロント(.ts/.tsx)に `service_role` 使用は無し（最も危険な漏れは不在）。definer関数群は `app_user_tenant_id()/auth.uid()/app_has_role()` でテナント・ロールを**auth由来で自己解決**（client偽装でない）設計。
- ただし未確認＝外販前に監査必須：(a) 全テーブルで RLS が ENABLE か、(b) すべての `SECURITY DEFINER` 関数がテナント/ロールで自己スコープしRLSを意図せず迂回していないか、(c) 全クエリで tenant_id/store_id 条件が効くか、(d) ロール(owner/admin/staff)とRLS条件が混線していないか。
- 結論：**「RLSは設定済み。ただし外販前にRPC含め実効確認が必要」**（安心と断定しない）。
