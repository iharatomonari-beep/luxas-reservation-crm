作成日: 2026-06-13

進め方: Cowork（司令塔）が1機能ずつ指示文を作る → Claude Code が実装＋自己検証 → 結果を持ち帰り次へ。1指示=1機能。原因切り分け→実装の順。

## 当面のフェーズ: 予約台帳コアの完成

予約台帳が安定しないと他機能（受付拡充・集計・DB移行）が積み上がらないため、ここを最優先で本番品質にする。

### タスク順（1つずつ Claude Code に投げる）

| # | タスク | 目的 | 状態 |
|---|---|---|---|
| T1 | ドラッグ移動の確認＆診断ログ整理 | 歴史的最優先課題の決着。コード実装済み＋診断ログ入りを、静的確認→1回実機確認→ログ削除で確定 | 着手 |
| T2 | 店舗設定マスタの導入 | `businessStart/End`・`slotMinutes`・受付開始/終了をハードコードからマスタへ。10時ずれ／営業時間固定の根治の土台 | 未 |
| T3 | 「10時ずれ」の根治 | `startTime` 未渡し時に必ず10:00へ落ちる挙動を、店舗設定の受付開始時刻参照 or 明示エラーへ | 未（T2依存） |
| T4 | 予約台帳ツールバー（カテゴリ絞り込み＋インターバル表示） | PM相当の台帳操作性。まず2機能から | 未 |
| T5 | 予約集計バー（台帳下部の当日集計） | 当日件数／スタッフ別／ステータス別など localStorage で出せる範囲 | 未 |
| T6 | 当日情報パネルの整理 | 「選択中の予約」を予約情報タブとして整理（販売/返客は枠のみ or 後回し） | 未 |
| T7 | 予約一覧テーブル画面 | 日付・顧客・スタッフ・コース・状態の一覧。台帳とは別ルート | 未 |

> T4〜T7 は着手時にスコープを再確認する（PMスクショとの突き合わせが必要な箇所は都度確認）。

## 全指示共通の「自己検証フッター」

毎回ブラウザで目視確認しなくて済むよう、Claude Code への全指示の末尾に以下を必ず付ける。

```
## 完了条件（自己検証・必須）
1. `npm run lint` がエラー0で通る。
2. `npm run build` が成功する（型エラー0）。
3. 変更したファイルと変更点を箇条書きで報告する。
4. 既存の予約台帳・予約作成・顧客管理・CSV入出力の挙動を壊していないか、関連コードを読み返して自己チェックする。
5. `docs/sessions/YYYY-MM-DD.md`（当日）に「やったこと／現在の状態／次にやること／既知の問題」を追記する。
6. 仕様で判断できない点があれば、勝手に決めず「確認が必要」として報告を止める。
```

UI挙動そのもの（ドラッグ等）はビルドで検証できないため、その種のタスクだけ「ユーザーが1回だけ実機確認」を挟む。それ以外は Claude Code の自己検証で進める。
