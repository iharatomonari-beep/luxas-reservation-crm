# LUXAS予約・顧客管理システム セキュリティ・個人情報保護方針

この文書は、LUXAS予約・顧客管理システム v0.1 の本番化前に、個人情報保護とセキュリティ設計の方針を整理したものです。

注意:
- この文書は設計方針であり、法的判断の最終確定ではありません。
- 個人情報保護法、関連ガイドライン、業界ルール、契約条件の最終確認は、必ず専門家の確認を受けてください。
- ここでいう「本番」は、顧客データを実際に運用する環境を指します。

## 1. 基本方針

1. 顧客情報、カルテ、注意事項、予約履歴、売上情報は、業務上必要な最小限の範囲で扱います。
2. 開発・検証用の `localStorage` は本番利用しません。
3. 本番DBは、Supabase PostgreSQL を候補として設計します。
4. 本番環境では、認証、権限分離、監査、バックアップ、復旧手順を前提にします。
5. 画面確認用のサンプルデータに、個人情報は入れません。

## 2. 保存方針

### 2-1. 開発・検証

- `localStorage` は、現行のプロトタイプ確認用に限定します。
- `localStorage` には本番顧客データを保存しません。
- ブラウザ依存のため、データ永続性、共有制御、監査、バックアップの点で本番用途に使いません。

### 2-2. 本番

- 本番DBは Supabase PostgreSQL を候補とします。
- 顧客、予約、売上、操作ログはDBで管理します。
- ファイル添付が必要になった場合は、保存先と権限を別途設計します。
- 取り込み用CSVやエクスポート用CSVは、一時ファイルとして扱い、保管期間を決めます。

## 3. データ種別と扱い

### 3-1. 顧客情報

顧客名、連絡先、住所、会員番号、タグ、ランクなどは個人情報として扱います。

扱い方:
- 画面では必要最小限を表示します。
- 参照権限を限定します。
- CSV出力は権限と目的を限定します。

### 3-2. カルテ・注意事項

カルテ、注意事項、備考、来店時のコメントは、業務上センシティブな内容を含む可能性があります。

扱い方:
- 予約担当と施術担当に限定して参照できる設計を前提にします。
- 画面表示は必要な箇所だけに限定します。
- CSV出力や画面コピーでの持ち出しを抑えます。

### 3-3. 予約履歴

予約履歴は、顧客の行動履歴として取り扱います。

扱い方:
- 予約履歴は顧客詳細や予約台帳で参照します。
- 参照には権限差を設けます。
- 不要な長期保存を避けるルールを決めます。

### 3-4. 売上情報

売上情報は、顧客単位・予約単位の業務情報として扱います。

扱い方:
- 管理者・責任者・会計担当など、限定された役割だけが参照できるようにします。
- CSV出力の対象を制限します。
- 集計や分析に使う場合は、匿名化や集約を優先します。

### 3-5. 要配慮個人情報に近い情報

健康状態そのものを直接保持していなくても、施術上の注意、体質、アレルギー傾向、肌状態、通院状況に近い記載が入る可能性があります。

扱い方:
- 要配慮個人情報に該当する可能性を想定して、保存・閲覧・出力を厳しめに設計します。
- 必要性がない項目は入力欄を増やしません。
- 画面上の説明文で、必要以上の自由記述を促さないようにします。

## 4. ロール別権限案

### 4-1. 想定ロール

- `owner`: 全権限
- `manager`: 店舗運営、顧客参照、予約操作、CSV管理
- `therapist`: 顧客参照、予約参照、カルテ参照、必要範囲で更新
- `reception`: 予約作成・編集、顧客の基本情報参照、電話受付向け操作

### 4-2. 権限の考え方

- 顧客詳細の全項目は、全ロールに見せない前提で設計します。
- カルテ、注意事項、売上、履歴、CSV出力は、役割により制限します。
- 削除、エクスポート、設定変更は上位ロールに限定します。
- 将来は画面単位ではなく、項目単位の閲覧制御も検討します。

## 5. CSV出力制限

- CSV出力は、基本的に上位ロールのみ許可します。
- 顧客明細、カルテ、注意事項、売上が含まれるCSVは、さらに厳格に制限します。
- エクスポート前に、対象件数と項目を画面で明示します。
- 画面からの一括ダウンロードは、監査ログを残します。
- メール添付や外部共有を前提にしません。
- 取り込み用のサンプルCSVと、本番データのCSVは分けて扱います。

## 6. 操作ログ方針

記録する操作の例:
- ログイン・ログアウト
- 顧客の作成、編集、参照
- 予約の作成、編集、キャンセル
- カルテや注意事項の更新
- CSVの取り込み、書き出し
- 権限変更
- エラーや失敗した操作

ログに残す項目:
- 実行日時
- 実行者
- ロール
- 操作種別
- 対象データ種別
- 対象件数
- 結果
- 必要最小限の識別子

ログに残さないもの:
- パスワード
- トークン
- 顧客の本文そのもの
- CSVの生データ全文

## 7. バックアップ方針

- 本番DBは定期バックアップを行います。
- 復元手順を事前に確認します。
- バックアップは世代管理します。
- バックアップの保管先は、本番DBと分離します。
- 復元テストを定期的に行います。
- 変更前にスナップショットを取る運用も検討します。

## 8. 漏えい発生時の初動対応

### 8-1. 直ちにやること

1. 影響範囲の拡大を止める。
2. 該当アカウント、APIキー、外部連携を一時停止する。
3. 漏えい経路を遮断する。
4. 証跡を保全する。
5. 事実関係を時系列で記録する。
6. 経営責任者と個人情報対応責任者に共有する。

### 8-2. その後にやること

1. 漏えい内容を分類する。
2. 件数、対象、期間、再発可能性を確認する。
3. 本人通知の要否を判断する。
4. 個人情報保護委員会への報告要否を判断する。
5. 必要に応じて公表や関係機関連絡を行う。
6. 再発防止策を作る。

## 9. 本人通知・個人情報保護委員会報告の要否確認手順

この確認は、必ず専門家の最終確認を前提にします。

### 9-1. まず確認すること

1. 漏えい等が「個人データ」に該当するか。
2. 漏えい等の対象に要配慮個人情報が含まれるか。
3. 財産的被害が生じるおそれがあるか。
4. 不正の目的をもって行われたおそれがあるか。
5. 影響件数が1,000人を超えるか。
6. 重大な二次被害のおそれがあるか。

### 9-2. 判断の進め方

1. 事実確認を先に行う。
2. 法務・顧問・専門家へ相談する。
3. 本人通知が必要か確認する。
4. 個人情報保護委員会への報告が必要か確認する。
5. 報告期限を確認する。
6. 必要なら公表の要否も検討する。

### 9-3. 参考になる公式情報

- 個人情報保護委員会「漏えい等報告・本人への通知の義務化について」

## 10. 本番前に必須にするセキュリティチェックリスト

### 10-1. 認証・認可

- [ ] 本番認証を有効化した
- [ ] ロール別権限を定義した
- [ ] 顧客・予約・売上の閲覧範囲を確認した
- [ ] CSV出力権限を制限した

### 10-2. データ保護

- [ ] localStorage を本番で使わない
- [ ] 本番DBを Supabase PostgreSQL に切り替える
- [ ] 個人情報の保存項目を最小化した
- [ ] カルテ・注意事項の扱いを確認した
- [ ] 退職者・権限変更時のアクセス停止手順を作った

### 10-3. 監査・運用

- [ ] 操作ログを保存する
- [ ] CSV取り込み・出力ログを残す
- [ ] バックアップと復元手順を確認した
- [ ] インシデント対応手順を定義した
- [ ] 連絡網を整備した

### 10-4. 画面・CSV

- [ ] 不要な個人情報を画面に出さない
- [ ] サンプルCSVに個人情報を入れない
- [ ] 本番CSV出力の項目を確認した
- [ ] 取り込み前プレビューで過剰表示しない

### 10-5. 法務確認

- [ ] 個人情報保護法上の義務を確認した
- [ ] 要配慮個人情報の扱いを確認した
- [ ] 本人通知・委員会報告の運用を確認した
- [ ] 必要に応じて専門家レビューを受けた

## 11. 運用メモ

- この方針は、実装が進むたびに見直します。
- 新しいCSV種別や外部連携を追加するときは、先にこの文書を更新します。
- 画面やDBを作る前に、保存目的と権限を決めます。
