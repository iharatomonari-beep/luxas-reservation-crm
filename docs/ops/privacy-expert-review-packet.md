# 個人情報保護 専門家確認用パケット（§5 #8）

作成: 2026-07-02 ／ 位置づけ: 実顧客名簿の投入・オンライン予約の本番公開前に行う「#8 個人情報の専門家確認」のための提出資料。
このまま弁護士・行政書士等の専門家に渡し、§6 の質問に回答をもらうことを目的とする。

---

## 1. 依頼の趣旨

整体サロン「LUXAS」の予約・顧客管理システム（自社開発・クラウド保存）について、
(1) 実顧客データの投入開始、(2) 顧客本人が使うオンライン予約ページの公開、(3) 将来の他社への外販（SaaS提供）
にあたり、個人情報保護法・関連ガイドライン上の問題点と必要な整備（規程・同意・契約）を確認したい。

## 2. 事業・システム概要

- 事業者: 株式会社東邦（整体サロン LUXAS を運営・国内7店舗をシステム登録済み）。
- システム: 予約台帳・顧客管理・会計・集計・メール配信（未稼働）・オンライン予約（顧客本人入力）。
- 構成: Web アプリ（Next.js/Vercel）＋ データベース（Supabase＝AWS 東京リージョン ap-northeast-1 上のクラウドサービス）。
- 現状: **実顧客データは未投入**（架空データで検証中）。スタッフ名簿・メニュー等の業務マスタは投入済み。
- 将来: 同システムを他社（他サロン企業）へ外販予定。マルチテナント方式（1つのDBに複数社を同居させ、行レベルセキュリティで分離）。

## 3. 取り扱う個人情報の項目（実装済みのフィールドから抽出）

- 顧客基本: 氏名・カナ、電話（最大3）、メール（最大2）、生年月日、性別、郵便番号・住所、職業、会員番号。
- 来店・取引: 予約履歴（日時・メニュー・担当・キャンセル歴）、来店回数、累計売上、支払方法別の会計記録。
- **施術関連（要注意）**: カルテメモ（chartMemo）、注意事項（caution）、施術タグ。**身体の状態・施術内容に関する記述が入り得るため、要配慮個人情報（健康情報）該当性の確認が必要（§6-Q1）**。
- 販促設定: DM/メール/プッシュの受け取り可否、ランク、タグ。
- オンライン予約時に本人が入力: 氏名・電話・メール・性別（＋予約時の端末種別を自動判定して記録）。
- 操作記録: 監査ログ（**個人情報そのものは記録しない設計**＝操作種別・件数・理由のみ）。

## 4. データの流れ

1. **取得**: 店頭・電話でスタッフが入力／オンライン予約ページで本人が入力／既存システム PeakManager からの名簿移行（CSV・予定）。
2. **保存**: Supabase（AWS東京リージョン）。開発用の localStorage には実顧客データを置かない方針（security-policy.md §2）。
3. **利用**: 予約管理・施術記録・売上集計・（将来）メール配信。
4. **第三者への提供**: 現状なし。EPARK/ホットペッパービューティー経由の予約取込（TORICOM連携）を検討中＝**提供/委託の整理が必要（§6-Q4）**。詳細は `docs/ops/toricom-hpb-integration-investigation.md`。
5. **委託**: クラウド事業者 Supabase（DB保管）、Vercel（アプリ配信）。GitHub にはコードのみ（実データなし）。

## 5. 講じている安全管理措置（2026-07-02 時点の事実）

- 認証: スタッフは本認証ログイン（許可リスト方式・リスト外は拒否＝fail-closed）。
- アクセス制御: 行レベルセキュリティ（RLS）を全40テーブルで有効化・テナントID必須（NOT NULL）。CSV出力・顧客統合はオーナー権限限定のRPC。
- 監査: 操作監査ログ（個人情報を含めない設計）。
- バックアップ: 日次自動バックアップ（保持7日・Proプラン）＋**復元テスト実施済み（2026-07-02・別プロジェクトへ復元し件数一致とセキュリティ設定維持を確認・テスト環境は削除済み）**。手順書 `docs/ops/supabase-backup-restore.md`。
- 顧客統合（名寄せ）: 物理削除でなく復元可能なソフトマージ方式。
- 開発運用: サンプルデータに個人情報を入れない／実データはコード管理（GitHub）に含めない。

## 6. 専門家に確認したい質問

- **Q1（最重要）**: 整体施術のカルテメモ・注意事項は「要配慮個人情報」（健康に関する情報）に該当するか。該当する場合、取得時の同意の取り方・オンライン予約フォームでの設計・既存名簿（PeakManagerからの移行分）の扱い。
- **Q2**: プライバシーポリシー・利用目的の特定と公表について、店頭掲示・Web掲載で満たすべき内容の雛型または必須項目。オンライン予約ページに置くべき同意チェックの要否と文言。
- **Q3**: PeakManager（現行システム）からの既存顧客名簿の移行は、利用目的の変更に当たらないか。移行時に本人通知や同意が必要か。
- **Q4**: EPARK/ホットペッパービューティーからの予約データ取込（および当社からの提供）は「第三者提供」「委託」のどちらで整理すべきか。必要な契約・記録義務。
- **Q5**: Supabase/Vercel（いずれも米国法人・データは AWS 東京リージョン保管）の利用は「外国にある第三者への提供」規制との関係でどう整理されるか。委託先の監督として何を確認・記録すべきか。
- **Q6（外販時）**: 他社サロンへ SaaS 提供する場合、各利用企業と当社の関係（当社＝取扱いの委託を受ける立場）の整理と、締結すべき契約（データ処理条項・安全管理措置の明示）。マルチテナント同居（RLS分離）で足りるかの考え方。
- **Q7**: 漏えい等発生時の個人情報保護委員会への報告・本人通知の要否判断フローと、当社規模で用意しておくべき初動手順。
- **Q8**: 保存期間・削除方針（退会・長期未来店の顧客データ）、本人からの開示・訂正・利用停止請求への対応手順として最低限整備すべきもの。
